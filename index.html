VirusTotal confirmation UX

Your requirement: “use VirusTotal API to tell the user to confirm malicious downloads.” The best UX pattern is: scan at upload, then gate downloads with a clear confirmation screen if detections exceed a threshold.

SCAN
Server submits file hash (or file) to VirusTotal; stores verdict + timestamp.
THRESHOLD
If detections ≥ your policy (e.g., 2+ engines), mark as “Suspicious”.
CONFIRM
User sees a warning and must explicitly confirm to download (log the confirmation event).
POLICY: CONFIRM REQUIRED

Password rules (secure registration)

Required: at least 9 characters, at least 2 special characters, and no spaces. This page includes a live password validator UI.

  • Client-side validation Great for UX, but you must also validate on the server and hash passwords (Argon2/bcrypt).
  • Rate limiting & lockout Prevent brute force; add MFA later if you want “vault-grade” positioning.
POLICY: 9+ chars, 2+ specials, no spaces

Guest inactivity deletion (2 months)

Unregistered users’ files should auto-delete after 2 months of inactivity. The UI below explains the policy in plain language and hints at how to implement it.

ACTIVITY
Update last_accessed_at on upload + any download/view events.
ABANDONED
If now - last_accessed_at ≥ 60 days, mark as abandoned.
DELETE
A scheduled job deletes objects + metadata; optionally email warnings for registered users.
Important
Frontends cannot delete server files by themselves.
You’ll need storage + a backend scheduler (cron/queue) to enforce this policy.

You said: “I don’t have disk drives.”

That’s normal—most cloud storage products don’t use local disks. They use an object storage provider (S3-compatible) and rent storage as needed. This page is a frontend; below is a realistic build path.

  • Storage layer (no disks needed) Use S3 / Backblaze B2 / Cloudflare R2 / Wasabi / MinIO. Your app stores files there.
  • Backend API Handles auth, metadata, expiration, and signed upload/download URLs. Keep VirusTotal key here.
  • Secure registration system Password hashing (Argon2/bcrypt), rate limits, and session/JWT storage.
  • Scheduled cleanup Cron/queue worker that deletes expired and inactive-guest files; updates database accordingly.
